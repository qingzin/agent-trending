# Agent工具使用与环境交互技术趋势报告（2025.1—2026.3）

**基于293篇学术论文的深度分析**

---

## 1. 执行摘要

2025年1月至2026年3月，Agent工具使用与环境交互领域经历了一轮从"能用工具"到"善用工具"再到"创造工具"的能力阶梯跃迁。基于对293篇学术论文的系统分析，这一时期的核心叙事可以概括为：**AI Agent正在从被动的工具调用者进化为主动的环境交互者——它不仅能选择和使用预定义工具，还能发现新工具、创造新工具、甚至重塑与环境交互的接口本身。**

**研究热度呈阶梯式爆发增长。** 月均论文产出从年初的6篇攀升至2026年2月的38篇高峰，增幅超过530%。增长曲线呈现三级阶梯：第一级（2025年1-4月，6-13篇）为基础方法探索期；第二级（2025年5-9月，14-20篇）为GUI Agent与工具调用范式确立期；第三级（2025年10月-2026年3月，14-38篇）为MCP生态爆发与工程化规模化期。

**五大核心趋势定义领域走向：**

- **趋势一：GUI Agent从实验走向产品级系统**——跨平台、跨设备的自主界面操作能力快速成熟，形成完整的训练-评估-部署技术栈；
- **趋势二：MCP（Model Context Protocol）生态的爆发式崛起**——统一的工具-Agent交互协议催生了新的基础设施层，重塑了工具发现、集成和调用的范式；
- **趋势三：工具使用的RL训练革命**——从模仿学习到在线强化学习的范式转变，使Agent能在真实环境交互中自主习得工具使用策略；
- **趋势四：深度搜索与信息寻求Agent的成熟**——多轮检索-推理-整合的闭环系统在效率和可靠性上取得关键突破；
- **趋势五：Agent-环境接口的重新定义**——从适应人类界面到构建Agent原生接口，从预定义工具集到动态工具发现与创造。

**产业影响方面**，Agent工具使用能力是决定AI从"问答系统"升级为"任务执行系统"的关键分水岭。其成熟将直接重塑三大产业方向：企业自动化（Web/GUI任务编排）、知识工作（深度研究与信息分析）、以及软件工程（代码生成-编译-测试-修复全链路）。本报告预判，**GUI自动化Agent和企业级工具调用Agent将在未来12个月内进入规模化商业部署阶段**。

### 2.2 驱动因素分析

**第一次跳跃（2025年5月，5→20篇）** 的驱动力来自三个方向的同步爆发：(a) GUI Agent的训练数据生成方法论成熟[ScaleTrack, Visual Test-time Scaling, 5月]；(b) 工具使用Agent的RL训练可行性被验证[Compiler-R1, O²-Searcher, 5月]；(c) MCP协议的提出催生了ScaleMCP等动态工具管理框架[ScaleMCP, 5月]。

**第二次跳跃（2025年10月，20→31篇）** 受益于三个技术共识的形成：(a) 真实MCP环境成为Agent训练和评估的标准[TOUCAN, TheMCPCompany, MCP-Flow, 10月]；(b) Web Agent从模式匹配进化到结构化工具抽象[WALT, BrowserAgent, 10月]；(c) 工具使用的可靠性和可观测性成为核心关注点[MICE, Butterfly Effects, ToolScope, 10月]。

**历史高峰（2026年2月，38篇）** 反映了领域在基础设施和方法论成熟后的全面工程化冲刺。论文覆盖了从环境合成[ScaleEnv, TermiGen, CLI-Gym]、到数据生成[ANCHOR, TerminalTraj]、到训练方法[RC-GRPO, ASTER]、到部署优化[DLLM-Searcher, W&D]、到理论分析[Information Fidelity, Coding Agents Theory]的完整技术栈。

### 2.3 与相关领域的关联

与同期Agent规划与推理领域（335篇）对比，工具使用与环境交互领域（293篇）在总量上略低，但在增长加速度上更高（从6篇到38篇，增幅530%，vs 7篇到39篇，增幅450%）。两个领域呈现强互补关系：**规划推理提供"怎么想"的能力，工具使用提供"怎么做"的能力**。值得注意的是，2025年10月后两领域的论文重叠度显著增加（如深度搜索Agent同时涉及推理策略和工具调用），表明**规划与工具使用正在融合为统一的Agent能力维度**。

## 3. 核心技术趋势深度分析

### 趋势一：GUI Agent从实验走向产品级系统

**一句话定义：** GUI Agent——能够直接操作图形用户界面完成任务的AI系统——正在从学术概念走向具备跨平台、跨设备、自主任务执行能力的产品级系统。

**演进脉络：**

- **基础能力构建期（2025年1-3月）：** [UI-TARS, 1月]开创端到端原生GUI Agent模型，直接从截图感知并执行操作；[ReachAgent, 2月]提出两阶段框架分离"页面到达"和"操作"子任务；[TRISHUL, 2月]解决跨平台GUI元素定位的泛化问题。

- **训练范式突破期（2025年5-7月）：** [ScaleTrack, 5月]统一GUI样本模板+回溯规划策略；[Visual Test-time Scaling, 5月]提出RegionFocus动态放大与图像即地图机制；[GUI-explorer, 5月]自主探索功能感知轨迹与无监督知识挖掘；[MobileGUI-RL, 7月]首次在真实在线环境中通过RL训练GUI Agent。

- **产品化工程期（2025年8月-2026年2月）：** [Mobile-Agent-v3, 8月]构建GUI-Owl基础模型；[ComputerRL, 8月]提出API-GUI统一范式支持大规模在线RL训练；[Ferret-UI Lite, 9月]通过数据混合和RL构建紧凑端设备Agent；[MAI-UI, 12月]全尺寸基础GUI Agent家族；[OmegaUse, 1月]通用GUI Agent支持移动和桌面自主执行；[ToolTok, 1月]将操作建模为渐进式工具使用序列。

**关键技术突破：**

1. **[ComputerRL, 8月]：** 提出API-GUI统一范式——将程序化API调用和视觉GUI操作统一在同一个动作空间中。核心洞察在于：机器Agent不需要像人类一样只能通过GUI操作计算机，它可以在需要精确控制时调用API，在需要视觉理解时操作GUI。这一范式配合大规模在线RL训练基础设施，使开源GUI Agent首次在复杂桌面任务上接近商用水平。

2. **[ToolTok, 1月]：** 颠覆了传统的坐标预测范式。传统方法要求Agent预测点击位置的精确像素坐标（容易出错且难以泛化），ToolTok将每个UI操作建模为一个"工具Token"——对应于特定UI元素上的特定操作。这将GUI交互从"低级视觉定位"提升为"高级工具选择"，显著提升了泛化能力。

3. **[CUA-Skill, 1月]：** 构建可复用技能库，将人类计算机交互的常见模式编码为结构化"技能"（如"登录账户"、"填写表单"、"下载文件"）。每个技能包含触发条件、执行步骤和验证标准。Agent在执行任务时首先检索可用技能，大幅降低了长程任务的决策复杂度。

**应用场景热度排名（GUI Agent内部）：**

| 排名 | 场景 | 论文数 | 趋势 |
|------|------|--------|------|
| 1 | 移动设备自动化 | 25+ | 持续高热 |
| 2 | 桌面/PC自动化 | 20+ | 快速增长 |
| 3 | Web导航与自动化 | 30+ | 持续高热 |
| 4 | 跨平台通用Agent | 10+ | 新兴快速增长 |

**成熟度评估：** 处于**工程落地初期/早期商业化**阶段。移动GUI Agent已达商用可靠性门槛，桌面Agent紧随其后。**置信度：很高。**

### 趋势二：MCP生态的爆发式崛起——工具交互的"USB接口"

**一句话定义：** Model Context Protocol（MCP）作为统一的Agent-工具交互协议，正在催生一个标准化的工具发现、注册、调用和管理生态系统，类似于USB协议对外设连接的革命性影响。

**演进脉络：**

- **协议引入与早期探索（2025年5-7月）：** [ScaleMCP, 5月]首次提出基于MCP的动态工具检索与自动同步存储系统，支持Agent自主添加和管理工具；[Making REST APIs Agent-Ready/AutoMCP, 7月]实现从OpenAPI规范自动生成MCP服务器。

- **生态爆发期（2025年10月-2026年1月）：** [TOUCAN, 10月]从500个真实MCP环境合成150万工具交互轨迹；[TheMCPCompany, 10月]构建基于MCP的通用工具调用基准；[MCP-Flow, 10月]自动化发现MCP服务器与合成训练数据；[ToolACE-MCP, 1月]提出历史感知路由管道从MCP工具泛化到Agent Web。

- **理论化与质量治理期（2026年2-3月）：** [Information Fidelity in Tool-Using LLM Agents, 2月]引入鞅分析框架分析MCP Agent的误差积累；[MCP Tool Descriptions Are Smelly, 2月]形式化工具描述缺陷的分类与影响；[Agentic SPARQL-MCP, 1月]探索MCP在联合知识图谱查询中的应用。

**关键技术突破：**

1. **[TOUCAN, 10月]：** 标志性的数据工程工作。从500个真实MCP服务器环境中，通过多教师模型生成和质量过滤流程，合成了150万条高质量工具交互轨迹。这不仅解决了开源社区缺乏大规模工具使用训练数据的核心瓶颈，更重要的是证明了MCP协议的标准化使得跨工具、跨领域的数据合成成为可能——无需为每个新工具单独设计数据收集管道。

2. **[MCP-Zero, 6月]：** 提出"主动工具发现"（Active Tool Discovery）概念——Agent不再被动地从预定义工具集中选择，而是在任务执行过程中主动发现并请求所需的新工具。这将Agent的工具使用能力从"封闭集选择"升级为"开放集发现"，是Agent自主性的重大飞跃。

3. **[Information Fidelity in MCP Agents, 2月]：** 首次为MCP Agent的可靠性提供了理论保证。通过鞅理论分析，证明了MCP工具链中的误差呈线性累积（而非指数爆炸），这意味着通过适当的误差修正机制，长链工具调用的可靠性是可控的。这一理论结果为工业部署提供了重要的设计指导。

**技术原理简述：** MCP的核心设计思想是将工具交互标准化为三个层次：(a) **工具描述层**——每个工具通过结构化的"工具卡"声明其功能、输入输出格式和使用约束；(b) **发现与路由层**——Agent通过语义匹配或依赖图找到与当前任务相关的工具；(c) **执行与反馈层**——标准化的调用接口和结构化的执行反馈。MCP的价值不仅在于技术标准化，更在于它使工具生态的规模化成为可能——任何人都可以构建MCP服务器，任何Agent都可以发现并使用这些工具。

**成熟度评估：** 处于**早期工程落地/快速标准化**阶段。MCP协议已被Anthropic等公司采纳为标准，开源生态快速扩展。**置信度：很高。**

### 趋势三：工具使用的RL训练革命

**一句话定义：** Agent的工具使用能力从依赖人工标注轨迹的模仿学习，转向在真实或模拟环境中通过强化学习自主探索和优化工具使用策略。

**演进脉络：**

- **在线RL探索期（2025年5-7月）：** [Compiler-R1, 5月]首创RL驱动的编译器优化Agent；[O²-Searcher, 5月]利用RL与本地模拟搜索环境训练搜索Agent；[MobileGUI-RL, 7月]在真实在线环境中通过RL训练GUI Agent；[GTA1, 7月]测试时缩放选择动作提案结合RL改进视觉定位。

- **过程奖励与工具交互RL（2025年8-10月）：** [Guiding VLM Agents with Process Rewards, 4月]推理时通过奖励模型过程监督引导动作选择；[ComputerRL, 8月]提出API-GUI统一范式的大规模在线RL训练；[SPORT, 4月]通过逐步偏好优化自主发现有效工具使用策略。

- **专门化RL框架涌现（2025年9月-2026年2月）：** [VerlTool, 9月]统一模块化框架支持多模态工具异步执行；[Tool-R1, 9月]通过RL生成可执行Python代码实现多步工具使用；[Process-Supervised RL, 9月]引入沙箱环境及轮次裁决RL；[ToolBrain, 9月]提供灵活RL框架支持自定义奖励；[RC-GRPO, 2月]解决多轮工具调用中奖励稀疏导致的优势信号消失问题；[ASTER, 2月]通过交互密集轨迹冷启动避免模型退化。

**关键技术突破：**

1. **[ComputerRL, 8月]：** 在GUI Agent领域实现了首个真正的大规模在线RL训练系统。核心创新在于：(a) API-GUI统一动作空间使RL优化的行为空间可控；(b) 自动化环境重置机制使RL rollout可以大规模并行；(c) 课程学习策略从简单任务逐步过渡到复杂任务避免训练崩塌。这一系统使开源GUI Agent首次通过纯RL训练达到了有竞争力的性能水平。

2. **[RC-GRPO, 2月]：** 解决了多轮工具调用RL训练中的一个关键难题——奖励稀疏导致"组内优势信号消失"。在标准GRPO中，同一批次内的多个轨迹如果奖励都相同（全部成功或全部失败），则无法计算相对优势。RC-GRPO引入奖励条件轨迹策略和离散奖励令牌，在组内维持多样性，确保优势信号始终可用。

3. **[ASTER, 2月]：** 发现了一个RL训练工具使用Agent的关键失败模式——"交互崩溃"。当RL扩展训练时，模型会退化为仅在核心推理完成后才进行工具调用作为"后验验证"，而非在推理过程中积极利用工具获取信息。ASTER通过交互密集的冷启动策略（在SFT阶段注入大量工具交互示例），避免了这种退化。

**成熟度评估：** 处于**原型开发后期/早期工程落地**阶段。RL训练GUI Agent已产出商用级模型，工具调用RL的通用框架正在形成。**置信度：高。**

### 趋势四：深度搜索与信息寻求Agent的成熟

**一句话定义：** 将Web搜索、文档检索和知识整合统一为Agent的核心工具使用能力，形成能够自主完成复杂信息寻求任务的成熟系统。

**演进脉络：**

- **基础搜索Agent（2025年1-3月）：** [Search-o1, 1月]将Agent式搜索嵌入推理链；[PaSa, 1月]LLM驱动的学术论文搜索Agent；[Open Deep Search, 3月]增强开源LLM的Web搜索推理能力。

- **深度搜索范式确立（2025年5-9月）：** [Deep Video Discovery, 5月]代理搜索策略自适应选择工具处理长视频；[Coding Agents with Multimodal Browsing, 6月]证明少量通用工具即可解决多样化任务；[From Web Search towards Agentic Deep Research, 6月]整合自主推理、迭代检索与动态反馈；[WebExplorer, 9月]系统化数据生成训练长程Web Agent；[InfoAgent, 9月]结合创新数据合成与Web搜索编排。

- **效率与可靠性优化（2025年10月-2026年3月）：** [BrowseMaster, 8月]规划-执行Agent对实现可扩展浏览；[TURA, 8月]结合RAG与代理工具使用；[PRISM, 10月]三Agent检索系统实现高精度高召回；[Nested Browser-Use Learning, 12月]解耦交互控制与页面探索；[AgentIR, 3月]推理感知检索联合嵌入Agent推理迹线；[W&D, 2月]通过并行工具调用扩展深度研究Agent效率。

**关键技术突破：**

1. **[WALT, 10月]：** 核心洞察——Web Agent不应该逐步操作UI元素，而应该将网站的"潜在功能"逆向工程为可调用的高级工具。例如，不是通过点击搜索框、输入关键词、点击搜索按钮三步操作，而是直接调用`search(query)`工具。这将Web交互从"低级UI操作"提升为"高级功能调用"，大幅降低了长程任务的失败率。

2. **[W&D, 2月]：** 发现深度研究Agent的效率瓶颈不在推理深度而在工具调用的串行性。提出"宽度扩展"策略——同时发起多个并行搜索查询，再汇总结果进行推理。通过优化"宽度"（并行搜索数）与"深度"（推理步数）的权衡，在保持结果质量的同时将延迟降低60%以上。

3. **[AgentIR, 3月]：** 提出"推理感知检索"——传统检索系统只匹配查询与文档的语义相似度，AgentIR将Agent的推理轨迹（reasoning trace）作为额外信号联合嵌入。这意味着检索系统不仅理解"Agent问了什么"，还理解"Agent为什么这样问"以及"Agent已经知道什么"，从而提供更精准的补充信息。

**成熟度评估：** 处于**工程落地/早期商业化**阶段。Perplexity、Google DeepResearch等产品已验证商业模式，学术界聚焦效率和可靠性的最后优化。**置信度：很高。**

### 趋势五：Agent-环境接口的重新定义

**一句话定义：** 研究社区正在从"让Agent适应人类界面"转向"为Agent构建原生接口"，同时发展Agent自主发现、创造和优化工具的能力。

**演进脉络：**

- **接口适应期（2025年1-6月）：** [GUI-Actor, 6月]提出基于注意力的无坐标视觉定位方法；[GOI, 10月]提出声明式原语分离策略与机制；[Position: Agent Should Invoke External Tools ONLY When Epistemically Necessary, 6月]提出工具调用的认知必要性理论。

- **Agent原生接口涌现（2025年7月-2026年2月）：** [Doc2Agent, 6月]可扩展管道从API文档生成可执行工具代理；[VOIX, 11月]通过声明式HTML标签暴露Agent可用动作与状态；[CI4A, 1月]语义封装机制将UI复杂交互抽象为工具原语；[Web Verbs, 2月]为Web动作提供类型化语义函数集。

- **工具创造与自进化（2025年2月-2026年3月）：** [ToolMaker, 2月]自主将带代码的论文转化为LLM兼容工具；[PLAY2PROMPT, 3月]通过"玩"工具自动探索输入输出行为生成文档；[Recon-Act, 9月]侦察团队对比错误与成功轨迹生成通用工具；[DIVE, 3月]逆向合成顺序——先执行工具再反推任务保证grounding。

**关键技术突破：**

1. **[ToolMaker, 2月]：** Agent不仅使用工具，还能创造工具。ToolMaker自主从学术论文中提取算法和代码，将其转化为LLM兼容的工具接口。通过闭环自校正调试（生成工具→测试→分析错误→修复），确保生成的工具可靠可用。这将Agent的工具生态从"人类预定义"扩展为"Agent自主扩展"。

2. **[VOIX, 11月]：** 代表了"为Agent构建Web"的思想。传统Web为人类设计——Agent需要从视觉渲染中"猜测"功能。VOIX通过声明式HTML标签（如`<voix-action>`、`<voix-state>`）直接在网页中标注Agent可用的动作和可观测的状态，使Web交互从"猜测+试错"变为"声明+执行"。如果这一标准被广泛采纳，将从根本上改变Web Agent的交互模式。

3. **[Position: Agent Should Invoke External Tools ONLY When Epistemically Necessary, 6月]：** 这篇立场论文提出了一个被广泛忽视的问题——Agent是否过度依赖工具？作者提出ToA（Theory of Agent）框架，认为Agent应仅在认知上确有必要时才调用外部工具，否则应依赖自身的内在知识。过度工具调用不仅效率低下，还会导致Agent内在推理能力的退化。这一观点对Agent系统设计产生了深远影响。

**技术原理简述：** Agent-环境接口重新定义的核心方向包括：(a) **从坐标到语义**——用语义化的工具/动作描述替代像素级坐标定位；(b) **从被动适应到主动声明**——环境通过标准化接口主动向Agent暴露可用功能；(c) **从静态到动态**——Agent能够在运行时发现、创建和优化工具；(d) **从人类视角到Agent视角**——接口设计从"人类可用"优先转向"Agent高效"优先。

**成熟度评估：** Agent原生接口处于**概念验证/早期原型**阶段；工具创造处于**原型开发中期**；接口标准化处于**早期标准化**阶段。**置信度：中高。**

### 趋势六：训练数据与环境的大规模合成

**一句话定义：** 自动化生成高质量工具交互轨迹、GUI操作数据和交互式训练环境，解决Agent训练的数据瓶颈。

**演进脉络：**

- **轨迹数据合成（2025年2-9月）：** [Explorer, 2月]大规模网络探索合成94K+多模态Web轨迹；[TOUCAN, 10月]150万MCP环境轨迹；[WebExplorer, 9月]系统化数据生成训练长程Web Agent。

- **环境合成（2025年10月-2026年3月）：** [WEBSERV, 10月]紧凑浏览器环境与可扩展RL环境；[OSGym, 11月]可扩展分布式OS副本环境引擎；[GTM, 12月]通用工具模型模拟工具执行；[GUI Exploration Lab, 12月]GUI仿真环境引擎；[EnvScaler, 1月]自动化框架通过程序化合成扩展环境规模；[ScaleEnv, 2月]从头构建fully interactive环境和verifiable任务；[TermiGen, 2月]合成verifiable环境和resilient expert轨迹；[CLI-Gym, 2月]利用Agent模拟环境历史反转生成任务。

**关键技术突破：**

1. **[ScaleEnv, 2月]：** 解决了一个根本性的鸡-蛋问题——训练Agent需要交互式环境，但构建交互式环境需要大量人工。ScaleEnv通过程序化合成从头构建完全交互式的工具环境和可验证的任务，无需人工设计单个环境。核心方法是利用LLM生成环境的API定义和状态转移逻辑，然后通过执行验证确保环境的正确性。

2. **[ANCHOR, 2月]：** 从种子演示（seed demonstration）出发，通过分支点变异（branch-point variation）大规模生成桌面GUI监督数据。核心思想是：在已有的正确轨迹中，在关键决策点引入变异（如选择不同的按钮、输入不同的文本），生成大量"接近正确但略有不同"的轨迹，极大地丰富了训练数据的多样性。

**成熟度评估：** 处于**原型开发后期/早期工程落地**阶段。数据合成管线已在多个开源项目中验证，环境合成框架正在标准化。**置信度：高。**

### 趋势七：工具使用的可靠性与效率治理

**一句话定义：** 针对Agent工具调用中的失败模式（参数错误、工具误选、过度调用、时间盲等），发展出系统性的检测、预防和恢复机制。

**演进脉络：**

- **失败模式分析（2025年5-7月）：** [Tool Preferences in Agentic LLMs are Unreliable, 5月]揭示工具描述编辑可drastically改变LLM工具选择；[Butterfly Effects in Toolchains, 7月]构建参数失败分类法分析工具Agent的参数填充失败。

- **可靠性增强机制（2025年9月-2026年2月）：** [PALADIN, 9月]训练Agent检测执行错误并从失败案例库恢复；[Your LLM Agents are Temporally Blind, 10月]揭示Agent"时间盲"缺陷并提出对齐技术；[RIMRULE, 12月]从失败轨迹蒸馏规则注入提示；[Recoverability Has a Law/ERR, 1月]形式化工具调用的可恢复性定律；[Confidence Dichotomy, 1月]联合优化任务准确性和校准。

- **效率优化（2025年10月-2026年2月）：** [Dynamic System Instructions and Tool Exposure, 12月]动态检索最小系统指令减少95%上下文token；[Optimizing Agentic Workflows using Meta-tools, 1月]识别并优化冗余工具执行模式；[Speculative Tool Calls, 12月]推测性工具调用优化推理延迟；[Budget-Aware Tool-Use, 11月]预算追踪器实现工具调用预算感知。

**关键技术突破：**

1. **[ERR/Recoverability Has a Law, 1月]：** 首次为工具调用Agent的自我恢复能力提供了数学理论。通过形式化"可恢复性"（Recoverability）指标ERR，推导出其与效率分数之间存在一阶线性关系——即Agent的恢复能力与其效率之间存在可预测的权衡。这一"恢复动力学定律"为Agent系统设计提供了理论指导：可以根据目标效率水平预估所需的恢复能力投资。

2. **[Dynamic System Instructions/ITR, 12月]：** 通过动态检索仅与当前任务相关的系统指令和工具描述（而非全量加载），将上下文token消耗降低95%。这解决了随着工具生态扩大，Agent的上下文窗口被大量无关工具描述占满的问题。

3. **[Your LLM Agents are Temporally Blind, 10月]：** 发现Agent在多轮交互中缺乏"时间感"——它无法准确感知两次工具调用之间经过了多长时间，因此无法判断之前获取的信息是否已经过时。这一发现对实时性敏感的应用（如金融交易、新闻核查）具有重要意义。

**成熟度评估：** 处于**原型开发中后期**阶段。个别可靠性机制已相对成熟，但系统性治理框架仍在形成中。**置信度：高。**

## 5. 技术架构演进分析

### 5.1 工具使用架构的三代演进

**第一代：函数调用（Function Calling）**
- 特征：LLM直接输出JSON格式的函数名和参数
- 代表：OpenAI Function Calling
- 局限：依赖准确的参数预测，错误率高，无反馈

**第二代：ReAct+工具（2025年上半年主流）**
- 特征：在思考-行动循环中嵌入工具调用，支持多步交互
- 代表：[OctoTools, 2月]、[TxAgent, 3月]
- 局限：工具选择依赖文本描述，缺乏全局规划

**第三代：Agentic工具系统（2025年下半年-2026年主流，当前最优范式）**
- 特征：
  - **工具发现层**：通过MCP/语义检索动态发现工具
  - **规划层**：全局规划工具调用序列和依赖关系
  - **执行层**：支持并行执行、异步反馈和错误恢复
  - **优化层**：RL训练优化工具选择和使用策略
  - **监控层**：预算感知、时间感知、可靠性监控
- 代表：[VerlTool, 9月]、[ASTER, 2月]、[Agent Alpha（搜索+工具）, 2月]
- 优势：灵活、可扩展、支持复杂任务

**萌芽的第四代：自进化工具生态**
- 特征：Agent能自主创建、优化和共享工具
- 早期信号：[ToolMaker, 2月]、[Recon-Act, 9月]、[DIVE, 3月]

### 5.2 GUI Agent架构的核心演进

```
坐标预测[UI-TARS] → 区域聚焦[RegionFocus]
                   → 技能抽象[CUA-Skill]
                   → 工具Token[ToolTok]
                   → API-GUI混合[ComputerRL]
```

### 5.3 工具检索与路由的演进

```
静态工具列表 → 语义检索[ToolScope]
             → 依赖图建模[Planning Agents on Ego-Graph]
             → 动态MCP发现[MCP-Zero]
             → 历史感知路由[ToolACE-MCP]
             → 自主工具创造[ToolMaker]
```

## 6. 挑战与瓶颈

### 6.1 被反复提出但尚未解决的核心问题

**第一位：工具选择的可靠性（>45篇论文提及）**
- 表现：Agent选错工具、传错参数、忽略必要工具
- 根因：工具描述的歧义性+LLM选择的不稳定性[Tool Preferences are Unreliable, 5月]
- 进展：[ToolScope, 10月]合并+过滤、[Natural Language Tools, 10月]去JSON化等方案部分缓解

**第二位：长程任务中的错误累积（>40篇论文提及）**
- 表现：单步小错误在多步执行中级联放大
- 根因：缺乏中间验证和错误恢复机制[Butterfly Effects, 7月]
- 进展：[PALADIN, 9月]失败案例恢复、[ERR, 1月]可恢复性理论化

**第三位：GUI Agent的视觉定位精度（>35篇论文提及）**
- 表现：在高分辨率复杂界面上无法准确点击目标元素
- 根因：视觉编码器的分辨率限制+UI元素的密集分布
- 进展：[RegionFocus, 5月]动态放大、[MEGA-GUI, 11月]多阶段粗-细定位

**第四位：训练数据的质量与多样性（>30篇论文提及）**
- 表现：合成数据噪声大，真实数据获取成本高
- 根因：交互式环境难以大规模构建[ScaleEnv, 2月]
- 进展：[WebSTAR, 11月]步级过滤、[ANCHOR, 2月]分支点变异

**第五位：计算成本与延迟（>25篇论文提及）**
- 表现：复杂任务需要数十次LLM推理和工具调用
- 根因：串行执行瓶颈+LLM推理成本
- 进展：[W&D, 2月]并行工具调用、[Speculative Tool Calls, 12月]推测性执行、[ITR, 12月]动态指令检索减95%上下文

### 6.2 结构性瓶颈

1. **环境碎片化：** Web环境、桌面环境、移动环境各有独立的训练和评估体系，跨环境迁移困难。[OmegaUse, 1月]尝试统一但尚未成为标准。

2. **安全与权限边界：** Agent在真实环境中执行操作（如点击按钮、填写表单、执行代码）存在不可逆风险。缺乏标准化的安全沙箱和权限控制机制。

3. **工具生态的碎片化与标准化博弈：** MCP虽然成为主流协议，但不同工具提供商的实现质量参差不齐[MCP Tool Descriptions Are Smelly, 2月]。工具描述的规范化是当务之急。

## 7. 未来展望与预测

### 7.1 未来12-18个月的技术走向预测

**预测一：MCP将成为Agent工具交互的事实标准（置信度：很高）**
- 类比USB协议对外设的统一作用，MCP将统一Agent-工具交互的接口规范
- 预计2026年底前主流LLM提供商将全部支持MCP调用
- 依据：[TOUCAN, 10月]已有150万MCP轨迹，[TheMCPCompany, 10月]提供标准基准

**预测二：GUI Agent将实现"一次训练，跨平台部署"（置信度：高）**
- 统一的跨平台GUI Agent基础模型将取代平台特定模型
- 预计将出现基于ToolTok等范式的通用GUI理解与操作模型
- 依据：[OmegaUse, 1月]、[ComputerRL, 8月]已展示跨平台可行性

**预测三：Agent将从"使用工具"进化为"管理工具生态"（置信度：中高）**
- Agent将具备工具创建、工具优化和工具推荐能力
- 类似于开发者管理软件包生态，Agent将管理工具生态
- 依据：[ToolMaker, 2月]、[Recon-Act, 9月]、[DIVE, 3月]展示了工具创造能力

**预测四：声明式Agent-Web接口将获得早期采纳（置信度：中）**
- 部分高频使用的Web服务将开始提供Agent优化的接口
- 预计先从API密集型场景（如SaaS工具、数据平台）开始
- 依据：[VOIX, 11月]、[Web Verbs, 2月]、[CI4A, 1月]提出了不同的接口范式

**预测五：工具使用的可靠性将通过形式化方法获得保障（置信度：中）**
- 基于[ERR, 1月]和[Information Fidelity, 2月]的理论基础，将发展出实用的可靠性验证工具
- 预计企业级部署将要求工具调用的可验证性保证
- 依据：金融、医疗等受监管行业的合规需求将驱动这一方向

### 7.2 可能的技术拐点

1. **"Agent Web"的出现：** 如果W3C或类似标准组织采纳Agent原生Web标准（如[VOIX, 11月]或[Web Verbs, 2月]提出的方案），将从根本上改变Web Agent的交互效率和可靠性。这是一个需要生态系统级别协调的拐点。

2. **工具使用的"Zero-shot泛化"：** 当Agent能够理解任意工具的功能并正确使用，而无需针对每个新工具进行训练时，Agent将获得真正的通用性。[Learning API Functionality from In-Context Demonstrations, 5月]和[SkillCraft, 2月]正在接近这一目标。

3. **Agent自主开发Agent：** 当Agent能够利用软件工程工具自主开发、测试和部署新的Agent系统时，将开启Agent能力的自我加速循环。当前的代码生成Agent[CompileAgent, 5月]和工具创造Agent[ToolMaker, 2月]是这一方向的早期信号。

| CUA | 计算机使用Agent | 能够使用计算机完成通用任务的Agent |
| Tool Calling | 工具调用 | Agent通过API或接口调用外部工具的能力 |
| Visual Grounding | 视觉定位 | 将语言描述映射到视觉空间中具体位置的能力 |
| DOM | 文档对象模型 | 网页的结构化表示 |
| Function Calling | 函数调用 | LLM输出结构化函数调用指令的能力 |
| ReAct | 推理-行动框架 | 交替进行推理和行动的Agent范式 |
| GRPO | 群体相对策略优化 | 多轨迹RL优化方法 |
| RAG | 检索增强生成 | 结合检索和生成的技术范式 |
| SFT | 监督微调 | 使用标注数据微调模型的方法 |
| RL | 强化学习 | 通过环境交互和奖励信号学习策略 |

| Rollout | 策略展开 | RL中按当前策略生成完整交互轨迹 |
